# 微调到底给工具调用 agent 带来了什么?

*一条小模型的能力主线 —— 从一个边际的 DPO 结果,到一个关于多步 agent 可靠性的架构发现。*

我想研究一个听起来很简单的问题:**当你把一个小开源模型微调成工具调用 agent 时,你究竟得到了什么?** 答案比"分数变高了"更有意思,也更诚实。而且它悄悄变成了一个关于**架构**、而非微调的结论。

这是整条主线的一份实战记录:我测了什么、哪里被显而易见的指标骗了、以及顺着这个"谎"摸到了更好的地方。所有东西都能从[仓库](.)复现(`README.md` 有入口)。

---

## 起点

一个有能力的开源基座(Qwen2.5-Instruct)开箱就能做真正的工具调用:读请求、判断要不要用工具、发出结构化调用、跑工具、读结果、循环直到能回答。微调本该让它*更好*。实验室来量化它;模型都在单卡上跑。

第一个结果干净且不意外:在函数调用数据(xlam)上做 LoRA-SFT,exact-match 工具调用准确率上升 —— Qwen2.5-7B 从 0.763 到 0.883,1.5B 从 0.703 到 0.810。增益几乎全在**参数**准确率上(模型本来就能选对工具、发出合法 JSON)。不错。然后呢?

## 发现 1 —— DPO 几乎没动,因为这局已经赢了

教科书式的下一级是偏好优化。我搭了完整的 DPO 闭环:把 SFT 模型**自己**的错误调用采样为"rejected"、金标调用为"chosen",训它偏好金标 —— 并让 SFT 模型既当策略起点、又当冻结参考。

结果:exact-match **0.883 → 0.890**。在噪声内。

直觉是"DPO 在这没用"。更真实的读法:**任务已经饱和了。** 在 500 条验证集、~0.88 的基线上,你根本测不出小于 2 个点的效应;而 DPO 这种"重塑分布"的方法又被贪心解码进一步盖住。教训不在 DPO —— 而在*要选一个有头部空间的指标*。于是我去找了一个。

## 发现 2 —— SFT 迁移的是*可靠性*,不是参数

如果 SFT 提升了参数准确率,它*泛化*吗?我在一个模型没训过的合成工具集(OOD)上评 base / SFT。

乍看 SFT **掉了**:exact-match 从 0.933(base)降到 0.820(SFT)。微调导致退步 —— 吓人。

然后我把实际错例打出来。**20 个里 20 个都是写法差异。** SFT 模型把金标的 `Japanese` 写成 `ja`、`3pm` 写成 `15:00` —— 语义正确、甚至*更标准*的重新序列化。它学会了把参数值规范化(ISO 代码、24 小时制),而这和字面字符串的金标对不上。

拆开看,真相反转了:SFT 把工具*选择*和格式可靠性推到了 ~1.0,**而且这迁移到了没见过的工具上**;所谓"参数退步"是指标假象。两个 takeaway:

- **exact-match 有一部分测的是"约定一致性",不是能力。** SFT 把约定拉向自己的训练分布,于是*虚高*了 in-distribution 分数、*虚低*了 OOD 分数。真实能力比这两个数都更稳。
- **所以 agent 要按执行来评** —— 这次调用*把任务办成了没有* —— 而不是跟参考字符串比对。

## 发现 3 —— 可靠性会连乘,然后这就成了架构的故事

我建了一个执行式、多步的 agent 评测。任务把真实工具串起来(锚在固定、猜不到的数据上,逼模型*必须*调用),并按任务有没有**办成**来打分,无关措辞或序列化。

在 1–3 步的简单任务上,微调后的 7B 从 0.90 跳到 **1.00** —— 而那个唯一干净的失败模式很说明问题。base-7B 在某一族任务("X 城的气温预计上升 D 度……")上反复*放弃*:它不去调 `get_weather` 把需要的值取回来,而是卡住("我需要当前气温")。SFT 恰好修了这个 —— 它让模型**主动去补齐缺失的信息**。可靠性,在多步上连乘。

接着我引入一个前沿开源模型来对比 —— 立刻撞上公平性问题(7B 对 20B 不公平)。正确答案不是 1v1,而是一条**尺度曲线**;而简单任务把所有强模型都顶饱和了(SFT-7B = 14B = gpt-oss-20b = 1.00)。于是我把任务**加难**:3–5 步链、一个干扰工具、比较逻辑。这把战场打开了:

| 模型 | 激活参数 | task_success(难) | 4步 | 5步 |
|---|---|---|---|---|
| Qwen2.5-7B base | 7B | 0.644 | 0.40 | 0.53 |
| Qwen2.5-7B + SFT | 7B | 0.778 | 0.90 | 0.50 |
| Qwen2.5-14B | 14B | 0.856 | 0.80 | 0.77 |
| gpt-oss-20b(OpenAI) | 3.6B(MoE) | **0.967** | 1.00 | 0.90 |
| Qwen3.6-35B-A3B(阿里) | 3B(MoE) | **0.944** | 0.83 | **1.00** |

三个结论层层递进:

1. **可靠性/深度墙真实存在。** dense 模型在最长链上崩(5 步 ≤ 0.77)—— 每步误差连乘(p^N)。
2. **"专精可抵规模" —— 但只成立一半。** SFT 把 7B 从 0.64 拉到 0.78,并在中等深度*追平*大模型(4 步 0.40 → 0.90)。但最深的链仍然要靠规模(5 步:SFT 0.50 vs 14B 0.77)。简单任务那个平局是饱和假象。
3. **真正的 takeaway 是效率 —— 而且它是架构层的。** 两个 ~3B 激活的 MoE 模型都登顶,*并双双越过 5 步深度墙*(0.90 / 1.00)—— 那道墙拦住了每一个 dense 模型,而它们只用约 1/4 的 per-token 算力。两家独立厂商 —— OpenAI 和阿里 —— 都复现了。**深度墙是 dense 架构的限制;稀疏 MoE + reasoning 用 ~3B 激活就破了它。** 单位激活算力下最强的长链 agent 可靠性,是一个*选哪种架构*的故事,不是*选哪家厂商*的故事。

## 我会下的结论,和我不会下的

- **我会下:** SFT 买到的是可迁移的*可靠性*(选工具、格式、主动性),而多步成功正是连乘在这种可靠性上;专精部分抵规模;稀疏 MoE+reasoning 模型在单位激活算力下拿到最强的 agent 可靠性,并独自挺过深链。
- **我不会下:** 说哪个模型"全面更强"(简单任务打平,排名只在压力下出现);说 DPO 没用(任务测不出它);说深度墙是根本性的(它是 dense 特有的)。
- **遗留问题:** 两个 Qwen(MoE 和 dense)都在同一个 C→F 换算族上栽 —— 一个具体、可命名、值得深挖的失败。而且这些任务是"查询之上的算术";更难的推理或强制错误恢复会压到不同的肌肉。

## 方法论笔记(可迁移的那部分)

- **饱和会掩盖一切。** 如果你的强基线都 ~1.0,你测的是任务、不是模型。下任何结论*之前*先造头部空间。
- **跟参考比对的 exact-match,是 agent 的陷阱。** 它惩罚合理的重新序列化、奖励约定匹配。按执行评。
- **把失败例打出来。** 这个项目里每一次真正的转弯都来自读原始输出 —— OOD 那个写法差异的"退步"、base 模型的放弃式失败、Qwen3.6 改了工具格式 —— 而不是来自聚合数字。
- **公平是一条曲线,不是一场决斗。** "A 比 B 强吗"跨尺寸最好用"扫尺寸直到差距闭合"来回答。
